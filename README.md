# HCAG Week 4: LLM-Based Architectural Recovery on Hadoop MapReduce

This repository contains the revised Week 4 implementation for **LLM-based architectural recovery** using an **HCAG-style hierarchical summarization pipeline**. The project applies a large language model to describe architectural components from clustered Java source files in the Hadoop MapReduce Client Core module.

Repository:

```text
https://github.com/zahediparsa/hpc_hcag_week4_revised
```

## Project Objective

The aim of this project is to generate high-level architectural descriptions for software clusters produced by different clustering algorithms. The target system is:

```text
Apache Hadoop MapReduce Client Core
```

The project applies the same LLM-based hierarchical summarization process to clusters generated by:

```text
ARC
ACDC
LIMBO
```

For each clustering algorithm, the pipeline produces a final CSV file with the required Week 4 format:

```csv
cluster_ID,files,title,description
```

Each cluster receives:

- an architectural title
- a concise architectural description
- the list of Java files contained in the cluster

The generated descriptions are intended to explain the cluster's components, interactions, quality attributes, and technologies used.

## What HCAG Means in This Project

HCAG refers to **Hierarchical Code/Architecture-Guided** abstraction. In this project, HCAG is used as a bottom-up summarization method for large source-code repositories.

Instead of passing all source code to the LLM at once, the codebase is summarized in layers:

1. **Leaf level: files**
   
   - Each Java source file is summarized from raw source code.
   - The file prompt extracts key functionality, core logic, inputs/outputs, and dependencies.

2. **Branch level: directories**
   
   - File summaries and already-generated subdirectory summaries are combined.
   - The LLM generates a higher-level directory abstraction.

3. **Cluster level: clustering algorithm output**
   
   - Directory summaries are combined into a final cluster-level architectural summary.
   - The final result describes the cluster's responsibility, component interactions, technologies, and quality attributes.

This approach is used because the full source code cannot fit into the LLM context window. Hierarchical summarization reduces the input size at each step while preserving architectural information across files and directories.

## RSF Preprocessing Notes

The ACDC and LIMBO clustering outputs were originally generated as RSF files. These RSF files were converted into file-level CSV files before running the LLM summarization pipeline.

During preprocessing, some RSF entities looked like Java file paths but did not correspond to real Java source files in the Hadoop MapReduce Client Core source tree. These invalid entries were manually removed before finalizing the CSV inputs.

The 24 removed entries were:

```text
jobhistory/EventType.java
jobhistory/Events.java
jobhistory/AMStarted.java
jobhistory/Event.java
jobhistory/JhCounter.java
jobhistory/JhCounterGroup.java
jobhistory/JhCounters.java
jobhistory/JobFinished.java
jobhistory/MapAttemptFinished.java
jobhistory/ReduceAttemptFinished.java
jobhistory/TaskAttemptFinished.java
jobhistory/TaskAttemptUnsuccessfulCompletion.java
jobhistory/TaskFailed.java
jobhistory/TaskFinished.java
jobhistory/JobInfoChange.java
jobhistory/JobInited.java
jobhistory/JobPriorityChange.java
jobhistory/JobQueueChange.java
jobhistory/JobStatusChanged.java
jobhistory/JobSubmitted.java
jobhistory/JobUnsuccessfulCompletion.java
jobhistory/TaskAttemptStarted.java
jobhistory/TaskStarted.java
jobhistory/TaskUpdated.java
```

These entries were not created as artificial source files. They were excluded to ensure that the LLM only receives real Java source files from the target Hadoop source directory.

A separate set of 18 real Hadoop source files existed in the Hadoop source tree but was not present in the ACDC and LIMBO RSF cluster outputs. These files were not added as singleton clusters in the final ACDC/LIMBO inputs. This preserves the original coverage of the ACDC and LIMBO clustering results after valid preprocessing.

## Model

The HPC run uses the Group 3 heavyweight model:

```text
ibm-granite/granite-34b-code-instruct-8k
```

The model is loaded from Hugging Face and requires an `HF_TOKEN` environment variable.

## HPC Environment

The Slurm script is configured for the PC2 Noctua HPC environment.

Default settings:

```text
partition: gpu
GPUs: 2 × A100
CPUs: 32
memory: 128 GB
time limit: 6 hours
Python module: lang/Python/3.11.3-GCCcore-12.3.0
```

## Reproducing the Run

### 1. Prepare the HPC project folder

```bash
export WORK_ROOT="/scratch/hpc-prf-dssecs/zahedi/ds4se_group3"
mkdir -p "$WORK_ROOT/code" "$WORK_ROOT/input"
```

You could change the username, however you'd have to update the slurm job accordingly.

### 2. Upload the code files

Place these files in:

```text
$WORK_ROOT/code/
```

Required files:

```text
week4_hcag_hpc_all_algorithms.py
week4_hcag_all_algorithms_run.sh
```

Make the Slurm script executable:

```bash
cd "$WORK_ROOT/code"
chmod +x week4_hcag_all_algorithms_run.sh
```

### 3. Upload the input CSV files

Place these files in:

```text
$WORK_ROOT/input/
```

Required files:

```text
clusters_arc.csv
clusters_acdc.csv
clusters_limbo.csv
```

### 4. Submit the Slurm job

```bash
cd "$WORK_ROOT/code"

read -rsp "HF token: " HF_TOKEN
echo
export HF_TOKEN

sbatch week4_hcag_all_algorithms_run.sh
```

### 5. Monitor the job

```bash
squeue -u USERNAME
ls -lt *.out *.err | head
tail -f ds4se_w4_hcag_all_<JOBID>.out
tail -80 ds4se_w4_hcag_all_<JOBID>.err
```

### 6. Result location on HPC

After completion, the output is written to:

```text
/scratch/hpc-prf-dssecs/USERNAME/ds4se_group3/week4_hcag_all_algorithms_output
```

To create a compressed archive for download:

```bash
cd /scratch/hpc-prf-dssecs/USERNAME/ds4se_group3
tar -czf week4_hcag_all_algorithms_output.tar.gz week4_hcag_all_algorithms_output
cp week4_hcag_all_algorithms_output.tar.gz ~/
```

Download from a local terminal:

```bash
scp USERNAME@fe.noctua2.pc2.uni-paderborn.de:~/week4_hcag_all_algorithms_output.tar.gz .
```

Extract locally:

```bash
tar -xzf week4_hcag_all_algorithms_output.tar.gz
```

## Notes on RQ2

The revised Week 4 instructions allow **zero-shot prompting** to be used for RQ2. This project therefore uses zero-shot structured hierarchical summarization prompts. No few-shot examples are included in the final pipeline.

The prompts directly instruct the LLM to generate:

- file-level semantic summaries
- directory-level architectural abstractions
- final cluster-level titles and descriptions

The final cluster prompts explicitly require:

- components and interactions
- quality attributes
- technologies used
- descriptions under 150 words

The generated RQ2 methodology report is available in:

```text
output/week4_hcag_all_algorithms_output/week4_rq2_zero_shot_report.md
```
